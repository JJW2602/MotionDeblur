## 1. Fine-tuning 실험 설계 (SDEdit / SD3 LoRA)

CFG guidance scale만으로는 deblurring이 안 되는 것을 확인. GoPro 데이터로 SD3 transformer LoRA fine-tuning을 진행하기로 결정.

### 1.1 3가지 Fine-tuning 방법

GPU 3개를 활용하여 아래 3가지를 병렬 실험:

| Task | 이름 | rank | alpha | LR | Text Encoder | 의도 |
|------|------|------|-------|----|-------------|------|
| 0 | `baseline_r8_lr1e4` | 8 | 8 | 1e-4 | 고정 | 기존 설정 + 개선된 logging으로 학습 여부 재확인 |
| 1 | `r32_lr1e4` | 32 | 32 | 1e-4 | 고정 | LoRA capacity 증가 (blur→sharp은 복잡한 구조적 변환) |
| 2 | `r16_lr5e5_textenc` | 16 | 16 | 5e-5 | 학습 | text encoder도 fine-tuning하여 deblurring prompt 이해 향상 |

**alpha = rank로 통일한 이유**: LoRA scaling factor = `alpha/rank`. 1.0으로 고정하면 rank만이 capacity를 결정하는 유일한 변수가 되어 실험 간 비교가 깔끔함.

**비교 전략**:
- 0 vs 1: rank 증가 효과 (capacity 부족 여부)
- 0 vs 2: text encoder 학습 효과 (conditioning 개선 여부)
- 1 vs 2: 높은 capacity vs text encoder 학습, 어느 쪽이 효과적인지

### 1.2 SLURM Script 업데이트

파일: `script/finetuning_sdedit.slurm`
- `--array=0-2` (GPU 3개, 각 1 method)
- partition: `base_suma_rtx3090`
- wandb_project: `Finetuning_LoRA_{METHOD_NAME}` (method별 분리)

## 2. Wandb Logging 대폭 개선

### 2.1 문제

기존에는 `train/loss` (raw loss)만 로깅. 매 step마다 다른 timestep을 샘플링하므로 loss scale이 달라 variance가 매우 커서, 실제 학습 여부를 판단할 수 없었음.

**핵심 인사이트**: timestep sampling 기법을 바꾸는 것이 아니라, logging을 개선하는 것이 해결책.

### 2.2 추가된 메트릭

| 패널 | 메트릭 | 용도 |
|------|--------|------|
| `loss/ema` | EMA (decay=0.99) smoothed loss | **학습 추이 확인의 핵심** — 우하향하면 학습 중 |
| `loss/ema_unweighted` | EMA unweighted MSE | weighting 영향 없는 순수 MSE 추이 |
| `loss/window_avg_100` | 최근 100 step 이동평균 | EMA보다 빠르게 반응, 단기 추이 확인 |
| `loss/raw` | 기존 raw loss | 참고용 (noisy) |
| `loss_by_t/low_t_avg` | timestep 하위 1/3 평균 loss | clean 영역 학습 확인 |
| `loss_by_t/mid_t_avg` | timestep 중간 1/3 평균 loss | 중간 noise 영역 |
| `loss_by_t/high_t_avg` | timestep 상위 1/3 평균 loss | 고noise 영역 |

### 2.3 Validation Loss 추가

- test split (129장) 전체에 대해 epoch 끝마다 평균 loss 계산
- `val/loss`: 전체 평균, `val_by_t/{low,mid,high}_avg`: timestep-bin별
- `torch.no_grad()`로 실행, train/eval mode 자동 전환
- **Overfitting 판단**: train EMA loss는 내려가는데 val/loss가 올라가기 시작하는 시점

## 3. Checkpoint 저장 방식 변경

### 3.1 기존
- 학습 종료 후 마지막 step의 weight만 저장

### 3.2 변경
- `save_every: 2000` step마다 중간 checkpoint 저장
- `max_train_steps`: 5000 → **10000** (~11 epochs)
- 파일명 패턴: `sd3_lora_{method}_step{N}.safetensors`

```
예시 (method=r32_lr1e4):

    sd3_lora_r32_lr1e4_step2000.safetensors
    sd3_lora_r32_lr1e4_step4000.safetensors
```

- val/loss 곡선과 비교하여 overfitting 직전의 checkpoint를 선택 가능

## 4. VRAM 사용량 분석

LoRA fine-tuning은 frozen model weight가 전체 VRAM의 ~80%를 차지하고, 이는 inference와 동일. LoRA 파라미터 + optimizer state는 수십~수백 MB 수준이므로 training과 inference의 VRAM 요구량이 비슷함.

| 방법 | 추정 VRAM |
|------|----------|
| baseline_r8 | ~12-13 GB |
| r32 | ~13-14 GB |
| r16 + textenc | ~13-15 GB |

RTX 3090 (24GB), A6000 (48GB) 모두 충분. 두 GPU 간 학습 속도 차이는 거의 없음 (동일 Ampere GA102, ~5-8% 차이).
